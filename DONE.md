## 2026-07-06 — Increment 5: live debugging (DBG-100/110/120/130/170)

* **Proto** (`proto/datapipe/runtime/v1/runtime.proto`): new `DebugChannel` bidi RPC on `RuntimeRegistryService` — the runtime pushes `DebugEvent`/`WireMetricsSnapshot` uplinks, the control plane pushes `SubscribeFlow`/`UnsubscribeFlow` downlinks so a runtime never captures or forwards data for a flow nobody is watching (DBG-170).
* **Engine** (`engine/flow/debug.go`, `debugctx.go`, `exec.go`): per-node ring buffer (default 50, DBG-100, always populated regardless of subscription) plus a separately rate-limited (default 20/s) live-forward path to an attached `DebugSink`; `flow.SidebarEvent` context-injected helper for node-authored sidebar pushes; `flow.ExecuteNode` runs one Processor node type in isolation against a pinned input (DBG-130), panic-safe (ARC-150). `engine/nodes/debuglog` upgraded from a console-only stub to the real DBG-110 debug node (selected-expression sidebar push, `console` now opt-in rather than the only behavior). Proven under a 20k-datagram burst test that the ring buffer stays capped and live-forwarded events stay near the rate limit while wire metrics still report the true, un-sampled volume (`TestDBG100_170_HighThroughputStaysBounded`) — the Increment 5 "done when" criterion.
* **Runtime client** (`engine/internal/runtimeclient/debugchannel.go`): `DebugSink` forwards only for subscribed flows; on `SubscribeFlow` immediately replays the flow's current ring-buffer snapshot (DBG-100 "inspection works without redeploy"); sends an empty "hello" as its very first uplink so the hub always has a runtime attached to send Subscribe/Unsubscribe to, even before any other uplink traffic exists — found and fixed via live browser verification (see below), the first version left a chicken-and-egg gap where a browser could subscribe before the runtime had any other reason to speak.
* **Control plane** (`controlplane/internal/debughub`): ref-counted per-flow subscriptions (Subscribe/Unsubscribe downlinks sent only on the first-subscriber/last-unsubscriber transition), a bounded per-flow cache for late-joining subscribers and "load full" lookups, payload truncation (4 KiB inline) before relaying to browsers. `controlplane/internal/api/debug.go`: WebSocket relay at `/ws/debug` (protocol documented spec-first in `docs/api/debug-websocket.md`, query-param token since the WS handshake can't carry a header, gated at Operator+ per DBG-170/SEC-110's "no payload inspection unless granted"), `POST .../nodes/{nodeId}/execute` (DBG-130, runs in-process via `engine/flow.ExecuteNode`, no runtime round-trip), pin CRUD (`debug_pins` migration) and `GET .../debug/events/{eventId}` (DBG-110 load-full). `docs/api/openapi.yaml` updated first per CLAUDE.md's spec-first rule.
* **UI**: `store/debug.ts` + `api/debugSocket.ts` (reconnecting WS client, per-node ring buffers, sidebar event list, wire pulse/metrics state); `components/JsonTree.tsx` (expandable tree, click-to-copy path/value); `components/Inspector.tsx` (new "Inspect" tab in `ConfigPanel`: live ring buffer in tree/raw view with load-full, run-once against a JSON payload, pin/unpin outputs); `components/DebugSidebar.tsx` (bottom-docked, filter by node, pause/clear); `FlowCanvas.tsx` decorates edges with a transient pulse class (DBG-120, keyed by source node+port so a fan-out pulses every downstream wire together) and a live delivered/dropped counter label (keyed by the full wire tuple, immune to sampling since it's driven by the periodic metrics snapshot, not per-event).
* **Two real bugs found during live browser verification** (not headless-test theater — a real control plane + runtime + Vite dev server, real gRPC, real WebSocket, real deployed inject→set→debug pipeline): (1) the debug-channel "chicken-and-egg" gap above; (2) a duplicate-event bug where a StrictMode remount (or any brief window with two overlapping `DebugSocket` connections) double-appended the same event into `nodeEvents`/`sidebarEvents`, causing React "duplicate key" errors — fixed by de-duplicating on the event's ULID before appending, with a regression test. Also fixed a `useSyncExternalStore` "getSnapshot should be cached" warning caused by `nodeEvents[nodeId] ?? []` allocating a fresh array every call when a node has no events yet — replaced with a module-level `EMPTY_EVENTS` constant.
* **Verified**: full Go workspace (`engine`, `controlplane`, `cli`, `sdk`, `tests`) builds/vets/passes `-race`, `golangci-lint`/`buf lint` clean; UI `tsc`/`oxlint`/`vitest` (34 tests, 8 new for this increment)/`vite build` all clean. * **`controlplane/internal/crypto`** (SEC-120): real envelope encryption — a random per-credential DEK encrypts the value; the DEK is wrapped under a versioned master KEK (`DATAPIPE_MASTER_KEY`), so a future KEK rotation only re-wraps DEKs, never the values.

Credentials are write-only by construction — no type or route ever pairs metadata with a decrypted value.

## 2026-07-06 — CI: first real GitHub Actions run, three genuine bugs fixed

The `ci.yml` workflow had never actually been observed running on GitHub Actions (only verified locally with matching tool versions). The first real run (Holger-triggered) failed, and three distinct root causes were found and fixed by watching each subsequent run via the GitHub REST API (job/step/annotation endpoints — no repo-admin rights needed for those, unlike raw log download):

* **`proto` job**: `buf generate` needs `protoc-gen-go`/`protoc-gen-go-grpc` on `PATH` (they're `local:` plugins in `buf.gen.yaml`), but the job only installed `buf` itself. Added `actions/setup-go` + `go install` for both, pinned to the versions in `proto/gen/go/go.mod`. `(14b7089)`
* **`go` job**: `golangci-lint-action@v6` doesn't support golangci-lint v2 (we develop against v2.12.2 locally, per `.golangci.yml`'s `version: "2"`) — confirmed by the run's annotations ("you must update to golangci-lint-action v7"). Bumped to v7. Also pointed `setup-go`'s `cache-dependency-path` at the go.sum files that actually exist (`proto/gen/go`, `engine`, `controlplane`) instead of the repo root, clearing an unrelated cache-restore warning. `(d34f7b9)`
* **`compose-smoke-test` job**: `docker compose build` failed — reproduced locally with a plain `docker compose build` after splitting the combined `up --build` step into separate `build`/`up` steps for failure-location visibility (`4957df5`). Root cause: `go.work` has listed `./tests` as a workspace member since Increment 1, but neither Dockerfile ever copied `tests/` into the build context, so `go build` inside the image failed with "cannot load module ../tests listed in go.work file". Added `COPY tests ./tests` to both Dockerfiles. `(e73114b)`

All four jobs (`proto`, `go`, `ui`, `compose-smoke-test`) confirmed green on commit `e73114b`.
