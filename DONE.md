## 2026-07-07 — Increment 9: edge runtime + fleet

* **Store-and-forward** (EDGE-130): new `engine/storeforward` package — a size- and time-bounded, disk-backed FIFO queue (`Open`/`Enqueue`/`Peek`/`Remove`, restart-durable by construction: its own index is rebuilt purely from directory listing on `Open`, no separate index file to go stale) plus `Drain` (retries the queue's head with the shared CON-130 reconnect backoff, forever, until ctx cancellation — never gives up and falls through like the ordinary `"retry"` policy does). New fifth `errorPolicy.onError` value, `"storeForward"` (`engine/flow.StoreForwardPolicy`, Flow-File-Format.md documented), wired into `nodeRunner`: a delivery failure durably enqueues instead of failing/discarding, and `Deployment.startNode` spawns exactly one drain goroutine per node (shared across every input-port goroutine for `MultiInputProcessor` nodes, and joined by the *same* WaitGroup as the node's ordinary receive loop(s) so a hot-redeploy restart of that node can never open a second, independent `Queue` over the same on-disk directory while the old one might still be mid-delivery — the exact object-identity-preservation lesson learned the hard way in Increment 8, applied proactively here). `Deployment.SetDataDir`/`RUNTIME_DATA_DIR` env var configure where queues live; unset disables durable queuing (falls back to `"fail"` with a warning, never silently drops).
* **Fleet enrollment** (EDGE-120/ARC-210 "per-device credentials"): `RegisterRequest` gained `enrollment_token`; a new `registry.DeviceStore` interface (`Authenticate`/`GroupOf`/`DeviceInfo`) is checked on every `Register` call when configured. Enrollment tokens are opaque 32-byte random values, only ever stored as a SHA-256 hash (`controlplane/internal/api.Store`, mirroring how session tokens are already stored) — the plaintext is returned exactly once, in the `POST /runtime-enroll-tokens` response. Once a `runtime_id` has enrolled with a token, every subsequent `Register` for that id must present the *same* token (rejected otherwise) — a genuine per-device credential, not just a one-time bootstrap. A `runtime_id` presenting no token at all is still accepted if it has never enrolled (the pre-Increment-9 walking-skeleton path stays working unmodified for local/dev single-server setups).
* **Fleet health** (EDGE-120 "inventory with health (online, CPU, memory, flow status, versions)"): `HeartbeatRequest` gained `cpu_percent`/`memory_bytes`/`flow_statuses`; new `engine/internal/procstats` samples this process's own CPU (via `syscall.Getrusage`, portable across darwin/linux, checked cross-compiling for `linux/arm64` too) and memory (`runtime.MemStats.Sys`) on every heartbeat. Live health stays in `registry.Service`'s in-memory `runtimeState` (never persisted — it's telemetry, not configuration); `GET /runtimes` (`registry.Service.ListRuntimes`) merges it with `DeviceStore`-sourced admin-configured metadata (group/display name/enrolled) for the fleet inventory view.
* **Runtime groups + deploy targeting** (UI-220): new `runtime_groups`/`runtime_enroll_tokens`/`devices` tables (migration `0004_fleet.sql`). `Flow-File-Format.md`'s pre-existing (Increment-3-era, previously unused) `runtimeAssignment.group` field now actually does something: `registry.Service.DeployFlow` gained a `targetGroup` parameter and only pushes to connected runtimes whose enrolled group matches (`""` = broadcast to all, preserving the old default); `DeployStream`'s automatic re-push-on-(re)connect (ERR-150) applies the identical filter, so a reconnecting/crash-recovering runtime never receives a flow reserved for a different group. `controlplane/internal/api.flows.go`'s deploy/rollback handlers resolve `runtimeAssignment.group` straight out of the flow's own parsed content (`flow.Parse`, no new column needed) and thread it through `Deployer.DeployFlow`. New REST surface: `GET/POST /runtime-groups`, `DELETE /runtime-groups/{name}`, `GET/POST /runtime-enroll-tokens`, `DELETE /runtime-enroll-tokens/{id}` (revoke), `PATCH /runtimes/{id}` (rename/assign group) — all System-Admin-gated, added to `docs/api/openapi.yaml` first.

## 2026-07-07 — Increment 8: triggered workflows

* **Trigger nodes and execution tracking** (ENG-100/ENG-130): `NodeTypeInfo` gained a `Trigger bool` flag (meaningful for `KindSource`); `http-in` is now marked as one (each webhook call becomes a tracked execution). `engine/flow/validate.go` enforces Flow-File-Format.md §7 rule 4 in full: a flow's entry nodes must be either all plain sources (mode `streaming`) or all trigger nodes (mode `triggered`), never mixed. New `engine/flow/execution.go`: `Tracker` implements per-flow concurrency limiting (`maxConcurrency`/`concurrencyPolicy: queue|reject`, a real semaphore with blocking-queue or immediate-reject semantics — `ErrConcurrencyRejected` lets `http-in` answer 429), execution timeout (`executionTimeoutMs`, marks `failed`/`timeout` and frees the slot), and completion detection via a pending-descendant counter keyed by the root datagram's `correlationId` (DGM-160) — the execution id *is* that correlation id, no separate id space invented. `ExecutionSink` (Waiting/Started/NodeEvent/Finished) is called for every event unconditionally, unlike `DebugSink` which is sampled (DBG-170) — DBG-140 requires "every execution recorded". `settings.concurrencyPolicy` added to Flow-File-Format.md/`engine/flow.Settings`.
* **Dead letters** (ERR-130): new `engine/flow/deadletter.go` (`DeadLetterSink`). A node whose error resolves to `"fail"`/`"discard"` (after retries) or whose datagram's TTL expired before processing (`nodeRunner.dropExpired`, closing a documented DGM-100 gap) is captured instead of silently dropped.
* **Flow-level error handler** (ERR-120): an unhandled (`"fail"`-policy) node error is published to `topics.DefaultBroker` under a reserved `"$errors/<flowId>"` topic (`flow.ErrorFlowTopic`, `"*"` wildcards to every flow via MQTT's `#`) — `"discard"`/`"errorPort"` are the flow author's own deliberate handling and do not publish. New `error-trigger` node (`engine/nodes/errortrigger`) subscribes and re-emits as a fresh tracked execution — an error-handler flow is just an ordinary triggered flow. `settings.errorFlow` (flow-level) falls back to a new project-level `defaultErrorFlow` (`projects.default_error_flow` column, `PATCH /projects/{id}`), resolved by the control plane at deploy time and pushed to the runtime via a new `DeployStreamResponse.default_error_flow` proto field. Honest scope note: cross-flow delivery only works when both flows run in the same `Deployment` today (see TODO.md — `engine/flow.Deployment` is still single-flow-per-runtime, a pre-existing gap this increment didn't have to close since the demo doesn't require it).
* **Stop-and-error node** (ERR-140): `engine/nodes/stoperror` always returns a `*flow.NodeError` (expression-resolvable message/code) — deliberately fails an execution through the same ERR-100 policy machinery as any other node error.
* **Re-run** (DBG-140): `Deployment.ReplayOutput` (re-run from the trigger's own recorded output, fanned out to its current downstream targets) and `.ReplayInput` (re-run from a failed node's recorded input, delivered directly into that node's own inbox) both start a fresh tracked execution tagged `reRunOf`. `Deployment.ReinjectDeadLetter` delivers a dead letter back into its node's inbox *without* starting a new execution (a dead letter may belong to an ordinary streaming flow).
* **Crash recovery** (ERR-150): `DeployStream`'s open now immediately re-pushes every currently-deployed flow to a (re)connecting runtime (`DeployedFlowsLister`) — closing a real, previously-unmet requirement ("runtime restart restores all deployed flows... automatically") that had nothing behind it before this increment. A runtime whose `EventChannel` disconnects has its still-`running`/`waiting` executions marked `crashed` (`MarkRuntimeExecutionsCrashed`).
* **New proto RPC** `EventChannel` (bidi, mirrors `DebugChannel`'s shape but never sampled): uplink `ExecutionEvent`/`DeadLetterEvent`, downlink `RunExecution`/`CancelExecution`/`ReinjectDeadLetter`. Runtime side: `engine/internal/runtimeclient.EventSink` (implements `flow.ExecutionSink`+`flow.DeadLetterSink`) and `DeploymentTarget`. Control-plane side: `registry.Service.EventChannel` (single-goroutine-per-direction, matching grpc-go's safety contract), `ExecutionStore`/`DeployedFlowsLister` interfaces (decoupled from `runtimev1`/`api`, same layering as `Deployer`/`ConnectionResolver`) implemented via small adapters in `cmd/controlplane/main.go`.
* **Control plane storage + REST**: migration `0003_executions.sql` (`executions`, `execution_node_io`, `dead_letters` tables; `projects.default_error_flow` column). New `GET /flows/{id}/executions` (filter by status), `GET /executions/{id}` (+ full node-I/O trace), `POST /executions/{id}/rerun` (`from: start|node`, Operator+), `POST /executions/{id}/cancel` (Operator+), `GET /flows/{id}/dead-letters`, `DELETE /dead-letters/{id}`, `POST /dead-letters/{id}/reinject` (Operator+) — all added to `docs/api/openapi.yaml` first.
* **UI**: new `FlowExecutions`/`ExecutionDetail`/`FlowDeadLetters` pages (`ui/src/pages/`), routed at `/projects/:id/flows/:id/executions[/:executionId]` and `/dead-letters`, linked from the flow editor header (Executions only for `mode: "triggered"` flows; Dead Letters for any flow). Follows the established local-`useState`-fetch-on-mount pattern (no new Zustand store), reuses `JsonTree` for per-node I/O, full en+de i18n coverage (UI-310).
* **A real, pre-existing bug found and fixed during live browser verification**: the runtime tagged every debug/execution event with the flow *content's own* `"id"` field (`Deployment.flowID = FlowFile.ID`), while the control plane and UI key every REST route and WebSocket/EventChannel subscription by the *database row's* id — two different strings whenever a flow's author-chosen content id doesn't happen to match the server-assigned one (always, in practice: the UI's "New flow" generates its own `flow_xxxxx` id independent of the DB UUID). This silently broke DBG-100/110/120 in the same way it broke Increment 8's new execution history, just never surfaced before now. Fixed at the one correct layer: `engine/cmd/runtime/main.go`'s `applyDeploy` now overwrites `ff.ID` with the control-plane-supplied id before calling `Deployment.Deploy`, making the runtime's internal `flowID` consistent with what everything else already keys by (CLI-only local deploys, which don't go through this path, are unaffected).
* **A second real bug found and fixed via the same live verification**: hot-redeploying a *single* node (e.g. fixing a script's bug) while a sibling node's own fingerprint was unchanged (so it wasn't restarted) caused every execution touching that unrestarted node to hang until the flow's `executionTimeoutMs` — `reconfigureTrackerLocked` was replacing `g.execTracker` with a brand-new `*Tracker` object on every `Deploy` call to pick up changed `maxConcurrency`/`concurrencyPolicy`/`executionTimeoutMs`, orphaning any node whose runner had already captured the *old* Tracker pointer at its own start time (the same "capture once, hold for the node's lifetime" pattern every other cross-cutting concern — `DebugSink`, `ConnectionResolver` — already relies on, just never violated internally before). Fixed with `Tracker.Reconfigure` (mutates settings in place, preserving object identity) instead of reconstruction; a regression test (`TestDBG140_ReplayInputRerunsFailedNodeAfterFix`) now asserts the re-run's execution reports `Finished` promptly rather than only checking data delivery, which is what let this one slip past the original test suite.
* **Demo flow**: `examples/webhook-divide-triggered.flow.json` (`http-in` → `script` → `http-response`, mode `triggered`) — matches the Development-Plan's "done when" line exactly, live-verified end to end via a real control plane + runtime + browser: webhook fired with `divisor:0` → node fails → execution history shows it `failed` with the script's error and a dead letter for the failing datagram → flow redeployed with a fix (`divisor === 0 ? null : ...`) → "re-run from this node" in the Executions UI → new execution completes `success` in ~0ms, `reRunOf` pointing at the original.

## 2026-07-07 — Increment 7: processor library P1

* **Expression engine** (`engine/expr`, MAP-130/150): one sandboxed JavaScript syntax used platform-wide (goja — no filesystem/network globals ever bound in, SEC-150), implementing Flow-File-Format.md §2's two documented conventions — whole-value `={{ expr }}` (`ResolveValue`/`ResolveDeep`) and mixed-template `{{ expr }}` splicing (`RenderTemplate`). Bindings: `payload`/`header`/`tags`/`env`/`flow`/`global` (the last two backed by `engine/ctxstore`, now wired into `engine/flow.Deployment` for the first time via `WithContextStore`/`SetContextStore`); stdlib globals `dt` (date/time with IANA time zones), `stats` (mean/stddev/percentile/etc.), `conv` (explicit type casting that throws rather than silently coercing, MAP-150), `hash` (md5/sha1/sha256). CPU/time limits via goja's `Interrupt` (ENG-150); a `Runtime`/`Program` pair lets hot-path nodes compile once and reuse a VM across every datagram. `engine/nodes/nodeutil.ExprData` builds the common `expr.Data` (payload/header/env/flow/global) once, shared by every node type below rather than duplicated nine times. Documented in full in `docs/Expression-Language.md`.
* **Engine architecture additions**, both proven necessary by real node needs rather than spec-mandated in the abstract: `flow.DynamicOutputs` (per-instance output ports depending on config, not fixed at type-registration — `switch`'s user-defined `out0..outN`, per Flow-File-Format.md §2's own documented port convention) and `flow.MultiInputProcessor` (named multi-input ports each fed from their own wire — `merge`'s two branches); both wired into `Deployment.startNode` and `Validate` (which now also instantiates every node during validation to ask instance-level questions, catching bad configs earlier). `ctxstore.Store` gained `Increment` (atomic, PROC-410's "atomic update operations"). `flow.AsNodeError`/`BuildErrorDatagram` exported for reuse outside the built-in error-policy path.
* **Flow-File-Format §7 rule 3** (node-config JSON-Schema validation) implemented: `engine/flow.Validate` now checks every node's config against its type's declared JSON Schema (`santhosh-tekuri/jsonschema/v5`, draft 2020-12) before attempting construction — closing a gap open since Increment 2.
* **Thirteen new processor/control node types**, each with its own unit test suite (several proven end-to-end under a real `flow.Deployment`, not just isolated `Process` calls):
  * **script** (PROC-100): inline JS with full datagram/header/tags access, `emit(port, value)` for multiple addressable outputs, `console.log` forwarded to the debug sidebar (DBG-110) instead of stdout, node-scoped `state.get/set` on top of the standard flow/global bindings — sandboxed and CPU/time-bounded like every other expression.
  * **convert** (PROC-120): JSON↔XML (a documented simplified subset — no attributes, array-under-key becomes repeated sibling elements), CSV↔records, base64, gzip, Latin-1↔UTF-8, and binary parse/serialize (scalar fields only, no bitfields yet).
  * **template** (PROC-130): `{{ expr }}` mixed templates — "logic-capable" because each placeholder is a full JS expression rather than a second template grammar.
  * **calculator** (PROC-200): ordered derived-field expressions, later fields see earlier ones' results, `stats.*` available for statistical derivations.
  * **window-aggregate** (PROC-210): tumbling/sliding/session windows by time or count, grouped by key expression, min/max/avg/sum/count/first/last/stddev/percentile/collect aggregates — closes and emits reactively on the next arrival after a boundary passes (documented architectural constraint, see TODO.md), not on a background timer.
  * **switch** (PROC-300): rule-based routing via boolean JS expressions (comparisons/regex/type-checks/quality/tag-matches are all just expressions, not a separate mini-language), first-match or all-matches, dynamic `out0..outN` + `default`.
  * **filter** (PROC-310): predicate pass/drop, or report-by-exception (deadband + max-interval heartbeat), optionally grouped.
  * **merge** (PROC-320): concatenate, combine-latest, windowed join (inner/left) by key expression, and correlationId batch-merge of two branches.
  * **split-batch** (PROC-330): split an array into singles, or collect singles into batches by count/time/byte-size.
  * **loop** (PROC-340): iterate array items one at a time over loop-back wiring, session state keyed by DGM-160's correlationId (not a custom tag, since ordinary `NewCaused` doesn't propagate tags through arbitrary intermediate nodes but does propagate correlationId) — guaranteed termination via a maxIterations cap, proven with a real loop-back wire under a live Deployment.
  * **try-catch** (PROC-370/ERR-110): wraps one other registered Processor node type via `flow.ExecuteNode`, routing its errors/panics to a "catch" port with the exact ERR-100 error-datagram shape — a deliberate scope reduction from a full visual multi-node error scope (UI-150 grouping doesn't exist yet), documented as such.
  * **lookup** (PROC-400): static table / SQL (reusing `sqlshared`) / HTTP sources, in-memory TTL + FIFO-eviction cache, fail/passthrough/default cache-miss policy.
  * **state** (PROC-410): node/flow/global-scoped read/get/set/delete/increment via `ctxstore`, values may be literal or `={{ expr }}`.

## 2026-07-06 — CI: first real GitHub Actions run, three genuine bugs fixed

Bumped to v7. Also pointed `setup-go`'s `cache-dependency-path` at the go.sum files that actually exist (`proto/gen/go`, `engine`, `controlplane`) instead of the repo root, clearing an unrelated cache-restore warning. `(d34f7b9)`
* **`compose-smoke-test` job**: `docker compose build` failed — reproduced locally with a plain `docker compose build` after splitting the combined `up --build` step into separate `build`/`up` steps for failure-location visibility (`4957df5`).
